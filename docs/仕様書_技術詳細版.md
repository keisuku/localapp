# Local App Studio 技術仕様書（セキュリティレビュー用）

対象読者: 情報システム部門、セキュリティベンダー、監査担当者
最終更新: 2026-06

---

## 1. システム概要

| 項目 | 内容 |
|---|---|
| アーキテクチャ | 完全クライアントサイド SPA（Single Page Application） |
| バックエンド | **なし**。API サーバー・DB サーバー・認証基盤を持たない |
| 外部通信 | **実行時ゼロ**（後述 §4。アプリコードに fetch/XHR/WebSocket/beacon の呼び出しが存在しない） |
| テレメトリ / 解析 | なし（Google Analytics 等の埋め込みなし） |
| 外部 CDN / Web フォント | 不使用。全アセットを自己ホスト（フォントは OS 標準スタックを使用） |
| 永続化 | ブラウザ IndexedDB（Same-Origin Policy によりオリジン単位で分離） |
| 主要依存 | React 19 / Vite 8 / Dexie 4 / SheetJS (xlsx 0.18) / TanStack Table 8 / zustand 5 / Radix UI |

## 2. データフロー

```
[ユーザーのファイル (.xlsx/.csv/.json)]
        │  ドラッグ&ドロップ / ファイル選択 / クリップボード貼り付け
        ▼
[File API → ArrayBuffer/Text]          … ブラウザサンドボックス内のメモリ
        ▼
[SheetJS によるパース]                  … 文字コード判定: UTF-8 厳格 → Shift_JIS フォールバック
        ▼
[列マッピング / バリデーション]          … 必須空欄・重複・範囲外数値・不正日付の検出
        ▼
[IndexedDB（DB名: local-app-studio）]   … Dexie 経由。全書き込みは単一モジュール
        ▼                                  (src/core/db/mutations.ts) を経由
[エクスポート]                          … Blob 生成 → ローカルダウンロードのみ
```

ネットワーク境界を越えるデータ移動は存在しない。すべての処理がブラウザのレンダラプロセス内で完結する。

## 3. 永続化仕様

### 3.1 IndexedDB スキーマ（Dexie v1）

```
records:       id, moduleId, status, *tags, updatedAt, archived, [moduleId+archived]
importHistory: id, moduleId, importedAt
savedFilters:  id, moduleId
settings:      key
```

- レコード本体: `{ id, moduleId, data: Record<string,unknown>, tags[], status, score, archived(0|1), createdAt, updatedAt }`
- ID は `crypto.getRandomValues` による 21 文字ランダム英数字（nanoid 相当、約 125bit）。
- 暗号化: IndexedDB 自体はアプリ層で暗号化しない。保存データの機密性は OS ユーザープロファイル境界とディスク暗号化（BitLocker / FileVault）に依存する。**機密データ運用時は端末のフルディスク暗号化を必須とすることを推奨**。
- 分離: Same-Origin Policy により、他サイト・他オリジンのスクリプトからはアクセス不可。

### 3.2 バックアップ形式

`{ version: 1, exportedAt, records[], importHistory[], savedFilters[], settings[] }` の単一 JSON。
復元時は `version` を検証し、確認ダイアログ → トランザクション内で全置換 → 直前スナップショットによる Undo を提供する。

## 4. 脅威モデルと対策

### 4.1 XSS（クロスサイトスクリプティング）

| 項目 | 状況 |
|---|---|
| `dangerouslySetInnerHTML` / `innerHTML` / `eval` / `new Function` | アプリコードで不使用（`grep` で検証可能） |
| 取り込みデータの描画 | すべて React の JSX 経由（自動エスケープ）。インポートした CSV/Excel/JSON の値は常にテキストノードとして描画される |
| URL フィールド | `<a href>` に `rel="noopener noreferrer"` `target="_blank"` を付与。`javascript:` スキームを入力しても href 文字列として描画されるのみで自動実行経路はない |
| 保存型 XSS | データソースが利用者自身のローカルファイルに限定され、かつ出力時にエスケープされるため成立しない |

### 4.2 CSV / 数式インジェクション

エクスポート時、CSV のセル値が `=` `+` `-` `@` または TAB で始まる場合に `'` を前置して数式評価を無効化する（`src/core/export/exportData.ts`）。xlsx 出力は文字列セル型として書き込まれ数式化されない。

### 4.3 プロトタイプ汚染

JSON / ファイル取り込み時に `__proto__` / `constructor` / `prototype` キーを除去する（`sanitizeData()`、`buildRowData()`、JSON ヘッダー抽出の 3 箇所で適用）。

### 4.4 依存パッケージ（サプライチェーン）

- `package-lock.json` によるバージョン固定。CI は `npm ci` を使用し lock と不一致なら失敗する。
- 実行時依存は §1 の主要ライブラリ + Radix UI プリミティブのみ。`postinstall` スクリプトを持つ依存はない。
- 推奨運用: 更新時に `npm audit` / `npm outdated` を実行し、差分レビューの上で lock を更新する。
- UI コンポーネントは shadcn/ui の生成物をリポジトリ内に同梱しており、外部レジストリへの実行時・ビルド時依存はない。

### 4.5 サービスワーカー（PWA）

`vite-plugin-pwa`（Workbox）による precache のみ。スコープは自オリジン配下、外部 URL のキャッシュ・ランタイムフェッチ書き換えは行わない。`registerType: 'autoUpdate'` により旧キャッシュは自動破棄。

### 4.6 残存リスク

| リスク | 評価 | 推奨対策 |
|---|---|---|
| 端末上の平文保存（IndexedDB） | 中 | フルディスク暗号化、端末ログインパスワード、共有端末での利用禁止 |
| ブラウザプロファイル消去によるデータ損失 | 中 | 定期 JSON バックアップの運用ルール化 |
| 配布ファイルの改ざん（オフライン版） | 低〜中 | 配布時に SHA-256 ハッシュを併記し受領側で検証（`certutil -hashfile` / `shasum -a 256`） |
| ブラウザ脆弱性経由の攻撃 | 低 | ブラウザの自動更新を有効化 |
| 利用者によるデータの二次流出（出力ファイル） | 運用課題 | 出力ファイルの取り扱い規程に従う |

## 5. 配布形態別のセキュリティ特性

| 形態 | 配信経路 | 完全性 | 備考 |
|---|---|---|---|
| GitHub Pages | HTTPS（TLS） | GitHub Actions による自動ビルド。ソースと成果物の対応が CI ログで追跡可能 | オリジン: `https://<owner>.github.io` |
| オフライン単一 HTML | 任意（ファイル配布） | ハッシュ検証を推奨 | `file://` オリジンで動作。IndexedDB はファイル配置場所のオリジンに紐づく |
| PWA インストール | HTTPS | Pages 版と同一 | Service Worker は同一オリジンに限定 |

## 6. ビルド・検証手順

```bash
npm ci                 # lock どおりの依存を取得
npm run typecheck      # TypeScript strict 検査
npm run build          # tsc -b && vite build && オフライン版生成・同梱
npm run preview        # 成果物のローカル確認
```

外部送信が無いことの検証方法（受け入れテスト例）:

1. ブラウザ DevTools → Network タブを開いた状態で全機能（取込・編集・出力・バックアップ）を操作し、初回ロードの静的アセット以外のリクエストが発生しないことを確認する。
2. `grep -rn "fetch(\|XMLHttpRequest\|sendBeacon\|WebSocket" src/` がヒットしないことを確認する。
3. オフライン単一 HTML 版を機内モードの端末で開き、全機能が動作することを確認する。

## 7. コード構成（監査の起点）

| パス | 役割 |
|---|---|
| `src/core/db/mutations.ts` | 全書き込みの唯一の経路（Undo 捕捉・監査ポイント） |
| `src/core/import/` | ファイル解析 → 型推定 → モジュール判定 → 検証 → コミット |
| `src/core/export/` | Excel/CSV/JSON 出力・バックアップ（CSV インジェクション対策含む） |
| `src/modules/` | 業務モジュール（スキーマ・スコアリング・ラベル）。コアから分離 |
| `.github/workflows/deploy.yml` | ビルド・デプロイのパイプライン定義 |
